## ✅ 推荐目录结构

💡 **最佳实践建议**：为实现高效、稳定且节省空间的媒体管理，请合理规划目录结构，并根据存储布局选择合适的文件链接方式。

### 1. 同一磁盘分区或同一文件系统（✅ 推荐）
在 Docker 环境中，若需支持**硬链接（hardlink）**功能，**媒体库目录与下载目录必须挂载在同一个文件系统（即同一挂载点）下**。否则，Docker 容器将判定目录跨分区，导致硬链接创建失败。

```bash
/Data/Media/               # 媒体库根目录
        ├── Movie/         # 电影目录
        └── Episodes/      # 电视剧目录
/Data/Downloads/           # 下载目录（与 Media 同属 /Data挂载点）
```

- ✅ **支持操作**：移动、复制、软链接、**硬链接**
- ✅ **节省空间**：可通过**硬链接或软链接**将下载完成的文件“接入”媒体库，无需重复存储
- ✅ **做种不受影响**：源文件仍保留在 `/Downloads/` 目录中，BT/PT 客户端可持续做种
- ✅ **推荐使用硬链接**：
  - 硬链接共享 inode，删除媒体库中的链接不会影响源文件（只要至少一个链接存在）。
  - 性能更优，且不依赖路径名，更加安全可靠。

> 🔍 **适用场景**：媒体库目录与下载目录在宿主机上（NAS）属于在同一硬盘或同一挂载分区（如 `/vol1/1000、存储空间1`），推荐此结构以最大化利用硬链接优势。

---

### 2. 跨磁盘分区或跨存储设备（⚠️ 可用但不推荐）

```bash
/Media/               # 媒体库根目录
   ├── Movie/         # 电影目录
   └── Episodes/      # 电视剧目录
/Downloads/           # 下载目录（与 /Media 属于独立的不同的挂载点）
```

- ✅ **支持操作**：移动、复制、**软链接**
- ❌ **不支持硬链接**：因硬链接要求文件与链接位于**同一文件系统**（inode 独立管理），跨分区无法创建
- ✅ **可节省空间**：通过**软链接**实现逻辑引用，避免物理复制
- ✅ **做种不受影响**：源文件仍在 `/Downloads/`，BT/PT 客户端可正常做种
- ⚠️ **注意事项**：
  - 软链接是“快捷方式”，若 `/Downloads` 分区未挂载、路径变更或源文件被移动/重命名，链接将失效（变为“悬空链接”）
  - 软链接稳定性较低，依赖路径一致性，维护成本较高

> 🔍 **适用场景**：仅当下载目录与媒体库目录不属于同一硬盘或同一挂载分区时使用（如： 下载目录位于`/vol1/1000、存储空间1`；媒体库目录位于`/vol2/1000、存储空间2`）。建议优先确保关键目录共处同一文件系统。

## 📂 目录配置说明

| 目录名称 | 说明 |
|-------|------|
| **源目录** | `源目录` 是需要监控和同步的目录（如下载目录）。程序会实时监控该目录中的新文件以及文件变化。 |
| **目的目录** | `目的目录` 是识别并重命名后存放的目标路径。也就是媒体库目录，程序会自动识别并按分类转移到 `媒体库目录`（电影、电视剧目录）。 |
| **未识别目录** | `未识别目录` 用于存放无法匹配影视信息的文件。建议默认值，需要人工干预进行处理。 |

## 🔗 转移方式详解

不同转移方式在性能、空间占用和做种影响方面各有差异，请根据需求选择。

### 1. **硬链接（Hard Link）**

- 一份数据生成多个文件入口，但**只占用一份磁盘空间**
- 只有当所有链接（包括原始文件）都被删除后，才会真正释放存储空间
- 可修改硬链接文件的名称，**不影响原文件及做种**
- ⚠️ **限制**：只能在同一文件系统（即同一磁盘分区）内创建，**不支持跨分区**
- ✅ **推荐场景**：本地单盘或多硬盘组成的同一存储空间（Raid），追求高性能与低存储开销

> 💡 提示：如您需要使用硬链接，请调整宿主机中下载目录、媒体库目录位置，移至同一文件系统的同一个文件夹内，并将该文件夹挂载到Docker容器内，否则会创建硬链接失败。

#### 硬链接原理图
```
[硬链接名1] } 
             } → 共享同一个 inode (123456) → 指向同一份数据块
[硬链接名2] }
```
- 两个名字指向同一份数据，删除任何一个，只要还有一个存在，数据就安全。

### 2. **软链接（Symbolic Link / Symlink）**

- 类似于 Windows 的“快捷方式”，仅包含指向目标文件的路径字符串
- 若原文件被删除、移动或重命名，软链接将失效（变成“悬空链接”）
- 创建时需确保目标路径正确且持久可用
- ✅ **支持跨分区、跨文件系统甚至网络路径**
- 在 Docker 环境下使用时，**容器内外的路径映射必须一致**，否则链接无法解析

> 💡 提示：在 Docker 中挂载目录时，请保持宿主机与容器内的路径结构一致，避免软链接失效。

#### 软链接原理图
```
[软链接] ---(指向路径)---> "/Downloads/movie.mkv"
                              ↓
                       [实际数据块] ← inode: 123456
```
- 如果 `/Downloads/movie.mkv` 被删除或移动，软链接失效。

**⚠️特别说明**
- 在 Docker 环境下使用时，**容器内外的路径映射必须一致**，否则链接无法解析
- Docker容器内创建的软链接在宿主机（NAS系统）上是无法解析和使用的，所以在NAS中只能看到这些文件，不能实际打开和播放这些文件；除非宿主机和容器路径完全一致。
- 不能跨系统使用，因为Docker容器内创建的软链接在宿主机上是无法解析和使用的，所以不能使用系统自带的媒体服务器应用（例如：极影视、飞牛影视、应用商店安装的套件版Emby、jellyfin等）

**⚠️使用软链接的容器内外路径映射必须一致，否则无法解析软链接**

### 3. **复制（Copy）**

- 完整复制一份文件副本到目标位置
- 文件数量翻倍，**额外占用等量磁盘空间**
- 源文件可独立删除或移动，不影响媒体库中的副本
- ✅ 不影响做种（前提是保留源文件）
- ❌ 存储效率低，适用于备份或归档场景

---

### 4. **移动（Move）**

- 将文件从源目录**物理迁移**至目的目录
- ✅ 节省空间，避免重复存储
- ❌ **会影响做种**：移动后原文件位置消失，BT/PT 客户端可能报错或停止做种

## ✅ 总结建议

| 场景 | 推荐方案 |
|------|----------|
| 单一硬盘，下载与媒体库同盘 | ✅ 使用 **硬链接** + 统一文件系统结构 |
| 多硬盘，下载与媒体库分离 | ✅ 使用 **软链接**，确保路径稳定、挂载可靠 |
| Docker 部署应用 | ✅ 宿主机与容器路径映射一致，避免软链接失效 |
| 追求极致做种稳定性 | ✅ 保留源文件，使用链接方式接入媒体库，避免移动 |

📌 **核心原则**：  
> **让源文件始终存在于下载目录中，通过链接方式组织媒体库，既能节省空间，又能保障做种连续性提升BT/PT客户端的下载速度。参与做种的客户端越多资源下载速度就越快**
